# Поиск по сайту: live-панель в шапке + страница /search

Добавлено 2026-07-04 (категории в выдаче — 2026-07-06). Миграций не потребовалось:
поиск работает по существующим таблицам под теми же RLS-политиками.

## 1. Что ищем и где

Поиск покрывает три сущности — регистронезависимая подстрока (`ilike`):

| Сущность | Поля поиска | Ссылка результата |
|---|---|---|
| категории (`categories`) | `name` | `/shop/[slug]` |
| товары (`products`) | `title`, `description`, `brand` | `/product/[slug]` |
| посты (`posts`, только `is_published` и `post_type='blog' — SEO-посты не выдаются) | `title`, `excerpt` | `/blog/[slug]` |

Полнотекстовый поиск (tsvector/pg_trgm) сознательно не заводили: каталог маленький,
`ilike '%q%'` его полностью покрывает. Если каталог вырастет и `ilike` по
`description` станет заметен — включить `pg_trgm` + GIN-индексы, интерфейс
функций (`lib/search.ts`) при этом не меняется.

## 2. Слой запросов — [lib/search.ts](../../../cozycorner/lib/search.ts)

Функции принимают Supabase-клиент аргументом (паттерн `fetchProducts`) и работают
и в браузере (панель в шапке), и на сервере (страница `/search`):

- `searchCategories / searchProducts / searchPosts(client, query, limit)` →
  `SearchResult<T> = { items, total }`. `total` — полное число совпадений
  (`count: "exact"`): им решается «показывать ли See all» и счётчики секций.
  Сортировки детерминированные, как у основных загрузчиков (вторичный ключ `id`).
- `searchPath(query)` → `/search?q=…` (encodeURIComponent) — единственный
  способ построить ссылку на страницу результатов.
- Лимиты: `SEARCH_DROPDOWN_LIMIT = 5` (панель), `SEARCH_CATEGORIES_LIMIT = 6`,
  `SEARCH_PRODUCTS_LIMIT = 24`, `SEARCH_POSTS_LIMIT = 12` (страница, без пагинации).

### sanitizeSearchQuery — почему это критично

Условия для нескольких колонок собираются строкой в PostgREST `.or()`:
`title.ilike.%q%,description.ilike.%q%`. Синтаксис этой строки ломается на
пользовательских `,` (разделитель условий) и `()` (группировка) — запрос
`sofa, cozy` без обработки давал бы 400 от PostgREST. Плюс `% _ \` —
спецсимволы LIKE. Санитайзер: обрезка до 100 символов → `,()` → пробел →
экранирование `\ % _` → схлопывание пробелов. Пустой результат санитайзера
(ввод из одних спецсимволов) вызывающие трактуют как «поиска нет».

## 3. Панель в шапке — [components/SearchBox.tsx](../../../cozycorner/components/SearchBox.tsx)

`'use client'`; заменил бывший декоративный инпут в
[Header.tsx](../../../cozycorner/components/Header.tsx) и владеет всем состоянием поиска.

- **Запуск**: ≥1 значимого символа → debounce 300 мс → три запроса параллельно
  (браузерный клиент из [lib/supabase/client.ts](../../../cozycorner/lib/supabase/client.ts)).
  Гонки давятся sequence-счётчиком в ref: применяется только ответ последнего
  запроса; cleanup эффекта инвалидирует ответы «в полёте». Ошибка Supabase
  ловится и показывает «ничего не найдено» — шапку не роняет.
- **Композиция строк** (суммарно ≤ 5): категории → товары → посты. Категории
  первыми — они навигационные и их обычно 1–2. Строка: миниатюра 44px
  (`ImageWithFallback`; у категорий cutout — `contain` на белой плитке, как в
  `CategoryCard`, у остальных `cover`) + название + мета (цена / `Category` /
  `Article`).
- **«See all N results»**: показывается при `total(категории+товары+посты) > 5`,
  ведёт на `/search?q=…`. Кнопка — full-bleed: отрицательные отступы гасят
  внутренний паддинг панели, у панели `overflow: hidden`, чтобы фон кнопки
  обрезался по скруглению.
- **Закрытие**: Escape (+blur), клик вне (`pointerdown` на document), переход на
  другую страницу (сброс по `usePathname()` через паттерн «adjust state during
  render» — setState в эффекте запрещён строгими правилами `react-hooks`).
  Enter → `router.push(searchPath(query))`.
- **Header**: пока панель открыта, auto-hide шапки выключен (проп
  `onOpenChange` → флаг `searchOpen` в Header, тот же механизм, что у
  мобильного меню).
- **Мобильный** (≤720px): инпут узкий (150px), но панель заякорена к правому
  краю (`right: 0; width: min(340px, calc(100vw - 2*var(--space-5)))`) — свешивается
  влево и в экран помещается.

## 4. Страница результатов — [app/search/page.tsx](../../../cozycorner/app/search/page.tsx)

Серверный компонент; `await searchParams` (Next 16: это Promise) сам делает роут
динамическим — `force-dynamic` не нужен. `robots: noindex, follow` — страницы
поиска не индексируем.

- **Hero** — собирается в коде по паттерну `/shop/[category]`: badge `Search`,
  title `Results for “query”` (обрезка ~60 символов), description — сводка
  `1 category · 8 products · 3 articles`, фон — от hero страницы `/shop`
  (`fetchPageContent(supabase, "shop")`), `align: left`.
- **Тело** — секции друг под другом, рендерятся только непустые, в порядке
  **Categories → Products → Articles**; заголовок секции со счётчиком полного
  числа совпадений. Сетка и Reveal-стагер — как в архиве блога. Всё сразу с
  лимитами из `lib/search.ts`, без пагинации/догрузки.
- **Состояния**: пустой/бессмысленный `q` → приглашение ввести запрос; ноль
  совпадений → «Nothing found … try a different search»; `?q=a&q=b` → берётся
  первое значение.

## 5. Как проверено

Playwright (системный Chrome) против dev-сервера, 2026-07-04/06 — всё прошло:

- панель: результаты через ~300 мс, ≤5 строк, корректные ссылки всех трёх типов;
  «furniture» → строка категории → `/shop/furniture`; «See all N results» при
  >5 совпадений, кнопка от борта до борта панели;
- `sofa, cozy (nice)` — ни одного 4xx от Supabase (санитайзер);
- Escape/Enter/клик-вне/переход закрывают панель, инпут очищается после навигации;
- `/search?q=kitchen` — смешанная выдача: сводка в hero + три секции по порядку;
  `/search` без `q` и с мусорным `q` — правильные пустые состояния; noindex в HTML;
- мобильный 390px: панель 340px, в экран помещается; auto-hide шапки не
  срабатывает при открытой панели;
- `npm run lint`, `npm run build` — чисто, `/search` собирается динамическим (ƒ).

## 6. Известные ограничения / на будущее

- **Релевантности нет**: сортировка выдачи — по дате (товары/посты), совпадение в
  `title` не приоритетнее совпадения в `description` (так, «bamboo» первым делом
  показывает товары, у которых бамбук лишь упомянут в описании). Лечится
  ранжированием (FTS `ts_rank` или простое «title-совпадения вперёд» на клиенте).
- **Короткие запросы**: по однобуквенному «a» все 5 строк панели занимают
  категории (они идут первыми). Если будет мешать — ограничить категории в панели
  1–2 строками.
- `ilike` нечувствителен к регистру, но не к диакритике — для английского
  каталога достаточно.
- SEO-поля (`seo_title`/`seo_description`) в поиск не включены намеренно — это
  служебные тексты.
