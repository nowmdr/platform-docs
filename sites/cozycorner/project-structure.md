# Структура проекта и конвенции

Держим код **компонентным и модульным**. Правила ниже — обязательны для нового кода.

## Дерево

```
app/                     # роуты (App Router)
  layout.tsx             # root layout: шрифт, метаданные по умолчанию
  page.tsx               # роут = только сборка данных + композиция секций
  globals.css            # дизайн-токены (:root) + сброс. Больше сюда почти ничего.
  <route>/page.tsx       # другие страницы по той же схеме
components/
  <Component>.tsx        # один компонент = один файл
  <Component>.module.css # его стили (CSS Module), рядом с компонентом
  <group>/               # связанные компоненты группируем в папку (напр. admin/)
  icons/                 # SVG-иконки как React-компоненты (Logo, SearchIcon, …)
lib/                     # не-UI логика: доступ к данным, хелперы, типы
  supabase/              # клиенты Supabase (client/server/admin) + storage
  <domain>.ts            # доменные функции (fetch/resolve/format)
  types.ts               # общие типы
supabase/
  migrations/            # версионируемые миграции (nnnn_name.sql), по порядку
  seed.sql               # тестовые данные
docs/                    # документация модулей и системы
```

## Правила

1. **Один компонент — один файл + свой `*.module.css` рядом.** Никаких инлайн-стилей и
   глобальных классов, кроме токенов в `globals.css`.
2. **Стили — только через токены дизайн-системы** (`var(--...)`). Новое значение →
   сначала токен, см. [design-system.md](design-system.md).
3. **Роут (`page.tsx`) тонкий**: грузит данные и композирует секции-компоненты.
   Разметку секций держим в компонентах, а не в `page.tsx`.
4. **Логика данных — в `lib/`**, не в компонентах. Компонент получает готовые данные
   пропсом (RSC) или зовёт хелпер из `lib/`.
5. **RSC по умолчанию**; `'use client'` — только там, где нужен браузер (состояние,
   observer, обработчики). Пример клиентского — `ProductGrid`.
6. **Группировка**: набор связанных компонентов одной фичи → папка в `components/`
   (как `components/admin/`) со своим `types.ts` при необходимости.
7. **Переиспользуемый элемент** (кнопка, бейдж…) при втором использовании выносится в
   отдельный компонент и попадает в каталог компонентов дизайн-системы.
8. **Изменения схемы БД** — только через новый файл миграции в `supabase/migrations/`
   и синхронизацию типов в `lib/` (см. [../../platform-docs/database/schema.md](../../database/schema.md) §7).
9. **Иконки — компоненты**, не картинки. Каждая SVG-иконка живёт в `components/icons/`
   отдельным компонентом, принимает `SVGProps<SVGSVGElement>` (`{...props}`), **без
   фиксированных `width/height`** (пропорции держит `viewBox`, размер — через CSS/
   `className`), а цвет — через `currentColor` (кроме брендовых цветов лого). Так иконки
   адаптивны: напр. лого меньше на мобилке. Исключение — `Logo` со своими фикс-цветами.

## Чек-лист перед коммитом фронта

- [ ] Нет хардкод-цветов/размеров — всё через токены.
- [ ] Новый UI-кусок оформлен компонентом со своим CSS Module.
- [ ] `page.tsx` не раздут разметкой — секции вынесены.
- [ ] Данные грузятся в `lib/`, компоненты не знают про SQL.
- [ ] Документация (design-system / этот файл) обновлена, если добавлен токен/компонент.
