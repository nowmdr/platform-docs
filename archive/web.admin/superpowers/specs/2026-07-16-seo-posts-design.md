# SEO Posts — дизайн (2026-07-16)

Новый раздел админки **SEO Posts**: посты, идентичные блогу по структуре, но не
попадающие в ленту блога на сайте. Назначение — programmatic SEO: контент под
long-tail-запросы, который создаёт агент (claude.ai + Supabase-коннектор), а
поисковики индексируют через sitemap. Утверждено в брейншторминге 2026-07-16.

## 1. Ключевые решения

- **Никакого cloaking'а.** Исходная идея «боту контент, человеку редирект на
  /blog» отклонена: любой механизм, отдающий ботам и людям разное, — прямое
  нарушение правил Google с риском санкций на весь домен. Вместо этого
  SEO-посты **доступны всем** по прямому URL (`/blog/[slug]` на сайте) и
  включены в sitemap, но **нигде не перечислены**: ни в ленте блога, ни в
  навигации. Живой человек попадает на них только из поиска — это и есть
  целевой посетитель.
- **Модель данных — вариант A**: колонка `post_type` в существующей таблице
  `posts` (не boolean, не отдельные таблицы). Максимальное переиспользование:
  таблицы секций, триггер slug, слой данных и оба компонента раздела Blog
  работают для обоих типов.
- **Название раздела** — «SEO Posts», путь в админке `/:siteSlug/seo-posts`.
- **Перенос поста между Blog и SEO Posts не делаем** (YAGNI): `updatePost`
  тип не меняет, UI переноса нет. Если понадобится — отдельная итерация.
- **Skill-файл** — markdown-документ `docs/seo-posts-skill.md` для загрузки в
  проект claude.ai (агент работает через Supabase-коннектор, не из этого репо).

## 2. База данных — миграция `add_seo_post_type`

Применяется к base-one через MCP (`apply_migration`), файл миграции затем
продублировать в репозитории сайта cozycorner (как `add_admin_folders`,
`add_content_folders`).

```sql
alter table cozycorner.posts
  add column post_type text not null default 'blog'
  check (post_type in ('blog', 'seo'));
create index posts_post_type_idx on cozycorner.posts (post_type);
```

- Существующие посты автоматически получают `'blog'` (default) — лента сайта
  не меняется до его обновления.
- RLS не трогаем: политики `posts` уже admin-only на запись, анон читает
  published — подходит и для SEO-постов (боты = анонимные читатели).
- `admin_folders` менять не нужно: CHECK-констрейнта на `section` нет
  (проверено в БД 2026-07-16), новая секция `'seo_posts'` работает сразу.

## 3. Слой данных — `src/lib/posts.ts`

- `export type PostType = 'blog' | 'seo'`.
- `Post` получает поле `post_type: PostType`; `post_type` добавляется в
  `POST_COLUMNS`.
- `listPosts(site, postType)` — обязательный второй аргумент, фильтр
  `.eq('post_type', postType)`. Существующий вызов в Blog передаёт `'blog'`.
- `createPost(site, input, modules)` — `post_type` входит в `PostInput`
  (форма проставляет его из пропа раздела). `updatePost` продолжает слать
  `PostInput`; поле в нём есть, но форма никогда его не меняет — тип
  фактически неизменен после создания.
- `getPost`/`deletePost` — без изменений (работают по id для любого типа).
- Query-ключи расширяются типом: список `['posts', site.slug, postType]`,
  пост `['posts', site.slug, postType, id]`. Инвалидация в форме — тот же
  паттерн `exact: true`, что описан в docs/blog.md §1.

## 4. Папки — `src/lib/folders.ts` + `src/features/folders/`

- `FolderSection` расширяется значением `'seo_posts'`.
- `SECTION_TABLES`: `seo_posts → 'posts'` (имя секции больше не всегда
  совпадает с именем таблицы — комментарий у константы обновить).
- У SEO Posts **свой набор папок** (section `'seo_posts'` в `admin_folders`),
  независимый от папок Blog. Поведение панели/чекбоксов/bulk-бара — без
  изменений (generic-код `features/folders/`).

## 5. Админка — UI

- `SiteLayout`: пункт **SEO Posts** сразу после Blog (иконка `Search` из
  lucide). Текст интерфейса — английский, как везде.
- Маршруты в `App.tsx`: `/:siteSlug/seo-posts`, `/seo-posts/new`,
  `/seo-posts/:postId`.
- **Новых компонентов нет.** `PostsPage` и `PostEditPage` принимают проп —
  конфиг-объект раздела (две константы в `features/posts/`): `postType`,
  базовый путь `blog`/`seo-posts`, folder section, лейблы («Posts» /
  «SEO Posts», «New post»). Маршруты Blog рендерят их с blog-конфигом,
  SEO Posts — с seo-конфигом. Поведение формы, модулей,
  SEO-фолбэка, гарда dirty, папок — идентично блогу.
- В списке и форме тип нигде не редактируется — он задан разделом.

## 6. Сайт cozycorner — промпт `docs/site-seo-posts-prompt.md`

Готовый промпт для агента репозитория сайта (по образцу
`docs/site-markdown-prompt.md`). Содержание:

- Лента блога (список постов): добавить фильтр `post_type = 'blog'`.
- Страница поста `/blog/[slug]`: фильтр по типу НЕ добавлять — пост любого
  типа рендерится одинаково по прямому URL.
- Sitemap: включать все published-посты обоих типов.
- Явный запрет: никаких редиректов для SEO-постов, никакого различения по
  User-Agent, никакого `noindex` — иначе фича теряет смысл или превращается
  в cloaking.
- Продублировать файл миграции `add_seo_post_type` в миграции сайта.

## 7. Skill-документ — `docs/seo-posts-skill.md`

Markdown-инструкция для проекта claude.ai, где агент с Supabase-коннектором
создаёт SEO-посты по команде вида «создай SEO-пост на тему X, вот похожий
пост». Перед написанием выполнить веб-поиск актуальных SEO-практик
(2025–2026) и отразить их. Структура документа:

1. **Зачем SEO-посты**: long-tail-запросы, programmatic SEO, страницы вне
   ленты; почему нельзя cloaking/редиректы.
2. **Контракт данных** (точный, чтобы агент не ломал инварианты):
   - insert в `cozycorner.posts`: `title`, `excerpt`, `post_type='seo'`,
     `is_published`, `published_at`, `cover_image_path` (плоский ключ бакета
     или внешний URL, можно null), `seo_title`, `seo_description`;
     **`slug` не отправлять** — генерирует БД-триггер; `content` (легаси) не
     трогать; `folder_id` опционально (null = Unsorted).
   - Секции: `post_text_sections` (`post_id`, `position`, `is_published`,
     `heading` nullable, `body` — markdown) и `post_product_sections`
     (`post_id`, `position`, `is_published`, `heading`, `columns` 2|3,
     `product_ids` uuid[]); **`position` сквозной** по объединению двух таблиц
     (0,1,2… в порядке чтения статьи).
   - `updated_at` нигде не задавать (триггеры).
3. **SEO best practices** (обязательный чек-лист для каждого поста):
   - **ВСЕГДА заполнять `seo_title` (≤60 символов) и `seo_description`
     (≤155 символов)** — уникальные для каждого поста, с целевым запросом;
     это жёсткое требование. («Теги» из исходной постановки = эти meta-теги:
     отдельного поля тегов у постов нет.)
   - Один пост = один поисковый интент; long-tail-формулировки в `title`.
   - Структура: заголовки секций (`heading`) как H2, логичная иерархия,
     ответ на запрос в первом абзаце.
   - Внутренние ссылки: products-модули с релевантными товарами, ссылки на
     существующие посты блога в markdown.
   - Качество: писать для людей (E-E-A-T), без keyword stuffing, без
     дублирования существующих постов; `excerpt` заполнять.
   - Дефолт публикации: создавать с `is_published=false` и включать после
     ревью человеком (агент может включить сразу только по явной команде).

## 8. Документация и проверка

- Правила раздела дописать в `docs/blog.md` (раздел про `post_type` и
  SEO Posts) — отдельный doc не заводим, код общий.
- CLAUDE.md: обновить статус.
- Проверка: `npm run lint`, `npm run build`; ручной e2e-чек-лист в плане
  (создание/редактирование SEO-поста, независимость папок, отсутствие
  SEO-постов в списке Blog и наоборот).

## 9. Вне скоупа

- Рендер на сайте (делает агент сайта по промпту §6).
- Перенос постов между типами.
- Поле тегов/рубрик у постов.
- Автогенерация контента из админки.
