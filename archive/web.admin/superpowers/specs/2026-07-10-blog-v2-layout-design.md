# Blog v2 + Products: раскладка редакторов, верхняя панель, фильтр статуса

Дата: 2026-07-10 (итерация по фидбеку после Blog v1). Утверждено пользователем.
Затрагивает: `src/features/posts/{PostsPage,PostEditPage}.tsx`,
`src/features/products/ProductEditPage.tsx`. Правила разделов —
`docs/blog.md`, `docs/products.md` (обновить после реализации).

## 1. Проблемы

1. В редакторах поста и товара левая колонка с картинкой занимает всю высоту
   страницы — под превью пустой столб.
2. В списке постов есть Draft-бейдж, но нет фильтра по статусу.
3. Save/Delete внизу длинной формы — чтобы сохранить, нужно проскроллить всё.

## 2. Решения

### 2.1 Верхняя панель редактора (пост И товар)

Одна строка на уровне ссылки «Back …»: слева ссылка назад, справа действия.
**Sticky**: панель прилипает к верху при скролле (`sticky top-0 z-10
bg-background` + нижняя граница/blur по вкусу реализации; проверить отступ
относительно шапки AppShell — если шапка не sticky, `top-0` достаточно).

- Пост: справа `Switch` Published/Draft (с текстовым лейблом) + `Save` +
  `Delete`. Поле Status из тела формы УБРАТЬ (не дублировать). Поле
  Published at остаётся в теле формы.
- Товар: справа только `Save` + `Delete` (статуса у товаров нет).
- Кнопка Save размещается вне тега `<form>` → сабмит через атрибут:
  `<form id="post-form">` / `id="product-form"` и `<Button type="submit"
  form="post-form">`. Тоггл — обычный Controller (RHF не требует нахождения
  внутри `<form>`).
- Состояния loading/error/not-found: рендерится только ссылка назад без
  кнопок (как сейчас — эти состояния живут в родительском компоненте
  страницы). Ссылка назад при отрендеренной форме переезжает в панель формы
  (родитель в этом состоянии свою не рендерит — не дублировать).
- Гард несохранённых изменений, поведение Save/Delete — без изменений.

### 2.2 Грид-шапка формы (картинка + ключевые поля)

Вместо текущего `flex md:flex-row` (aside на всю высоту) — форма становится
одной колонкой; сверху грид `md:grid-cols-[16rem_1fr] gap-6`:

- **Пост**: слева превью cover (квадрат, как сейчас) + slug под ним; справа —
  Title, Excerpt, Cover image (инпут + Gallery). Ниже грида на всю ширину:
  Published at (пол-ширины, `max-w-*`), Content (модули), SEO.
- **Товар**: слева превью + slug; справа — Title, грид Price+Brand,
  Image path (инпут + Gallery). Ниже на всю ширину: Referral URL,
  грид Category+Image style, Description (markdown), SEO.
- Ограничение ширины `max-w-xl` у правой части формы снимается для нижних
  секций разумно: сохранить читаемую ширину (например `max-w-3xl` на весь
  контейнер формы) — на усмотрение реализации, главное убрать пустой столб
  под картинкой.
- Мобильная раскладка: грид схлопывается в одну колонку (картинка сверху).

### 2.3 Фильтр статуса в списке постов (PostsPage)

По паттерну фильтров ProductsPage:

- `Select` шириной `w-44` рядом с поиском: `All statuses` (сентинел
  `__all__`) / `Published` / `Draft`. Клиентская фильтрация по
  `is_published` (`published` ↔ true, `draft` ↔ false).
- `hasFilter` учитывает и поиск, и статус; счётчик `видимые / все`;
  ghost-кнопка Clear сбрасывает оба.

## 3. Вне скоупа

- Список товаров, медиа — не трогаем.
- Порядок/состав остальных полей форм — без изменений.
- Sticky-панель для других разделов (media details и т.п.) — нет.

## 4. Проверка

`npm run lint && npm run build`; ручная проверка раскладки на md/mobile
ширинах, sticky при скролле длинного поста, фильтр Draft после переключения
статуса поста.
