# Промпт для агента репозитория сайта CozyCorner: рендер Markdown в description

Скопируй текст ниже в сессию агента, работающего с репозиторием сайта cozycorner.

---

Задача: научить сайт рендерить Markdown в поле `products.description`.

Контекст:
- Админ-панель теперь сохраняет `cozycorner.products.description` в **Markdown**
  (ограниченное подмножество): абзацы разделяются пустой строкой, маркированные
  списки `- пункт`, нумерованные `1. пункт`, выделение `**жирный**` и `*курсив*`.
  Заголовков, ссылок, картинок, таблиц, кода в этом поле НЕ бывает (редактор
  админки их не создаёт).
- Старые записи — плоский текст без разметки; валидный Markdown сам по себе,
  отдельный фолбэк не нужен.
- Поле опциональное (может быть NULL).

Что сделать:
1. Найди все места, где выводится `product.description` (страница
   `/product/[slug]`, возможно карточки/секции), и замени вывод плоского текста
   на рендер Markdown.
2. Используй `react-markdown` (без плагинов, HTML в контенте запрещён по
   умолчанию — так и оставить). Разрешённый набор элементов можно ограничить
   пропсом `allowedElements`: `p`, `ul`, `ol`, `li`, `strong`, `em` (+ `br`);
   остальное отключить (`unwrapDisallowed`).
3. Стилизуй результат под текущую типографику страницы товара: отступы между
   абзацами, маркеры/отступы списков (если Tailwind — например `[&_ul]:list-disc
   [&_ul]:pl-5 [&_ol]:list-decimal [&_ol]:pl-5`, или typography-plugin, если он
   уже используется).
4. Если description участвует в SEO/OG-описаниях или где-то обрезается до
   сниппета — там markdown-разметку нужно СТРИПАТЬ до плоского текста, а не
   рендерить (и не показывать `**` в мета-тегах).

Критерии приёмки:
- Товар с markdown-описанием (списки, два абзаца, жирный текст) рендерится
  корректно; сырые `-`, `**`, `1.` на странице не видны.
- Товар со старым плоским описанием выглядит как раньше (один абзац / переносы).
- Товар без описания — как раньше (блок не ломается).
- В мета-тегах (description/OG) — только плоский текст без разметки.

Примечание: поле `posts.content` блога пока НЕ переводим на markdown — это
отдельная задача, но выбранный подход (react-markdown + общий компонент)
должен легко переиспользоваться для блога.

# Дополнение (2026-07-10): тело статических страниц из `pages.body`

Задача: рендерить тело страниц terms/privacy (и других статических страниц,
если у них появится текст) из БД вместо захардкоженного в коде.

Контекст:
- В `cozycorner.pages` добавлена колонка `body text` (nullable). Миграция УЖЕ
  применена к живой БД из админки (через Supabase MCP, имя `add_pages_body`) —
  в репозиторий сайта нужно добавить файл миграции с тем же содержимым, чтобы
  история миграций осталась полной:
  ```sql
  alter table cozycorner.pages add column if not exists body text;
  comment on column cozycorner.pages.body is
    'Тело страницы в Markdown (редактируется админкой). NULL — сайт показывает свой встроенный текст.';
  ```
  `if not exists` делает реплей на живой БД безопасным.
- Формат `body` — тот же Markdown-подмножество, что в `products.description`
  (см. выше): абзацы, списки, **bold**/*italic*; без заголовков/ссылок/кода.
- Контракт: `body IS NULL` (или пусто) → страница показывает свой текущий
  захардкоженный текст (ничего не ломается); `body` заполнен → рендерить его
  вместо захардкоженного.

Что сделать:
1. Добавить файл миграции (SQL выше) в каталог миграций репозитория.
2. На страницах terms и privacy читать `pages.body` по slug страницы и, если
   он непустой, рендерить его тем же markdown-компонентом, что и
   `products.description`; иначе — текущий контент.
3. Ничего не менять на страницах, у которых тела нет (home/shop/blog/...) —
   для них `body` останется NULL.

Критерии приёмки:
- При NULL `body` terms/privacy выглядят ровно как сейчас.
- Заполненный из админки markdown-текст рендерится корректно (абзацы, списки,
  жирный), сырой разметки не видно.

# Дополнение 2 (2026-07-10): страницы не подхватывают изменения из БД

Симптом (проверено на живом сайте): в `cozycorner.pages.body` лежит новый
текст, а `/privacy` и `/terms` продолжают отдавать старый — страницы
отрендерены статически при билде и БД больше не перечитывают.

Задача: сделать так, чтобы правки из админки появлялись на сайте без
редеплоя.

Что сделать:
1. В route-сегментах страниц terms и privacy включить ISR:
   `export const revalidate = 60;` (страница перерендеривается не чаще раза
   в 60 секунд — для юридических текстов достаточно; при желании 300).
   Альтернатива `export const dynamic = 'force-dynamic'` — не нужна, это
   лишний рендер на каждый запрос.
2. Убедиться, что запрос к Supabase в этих страницах не закэширован жёстче
   сегмента (не передавать в fetch/клиент `cache: 'force-cache'` и свои
   `next.revalidate` длиннее сегментного).
3. Проверить остальные страницы, читающие контент из БД (каталог, блог,
   hero): если они тоже полностью статические без revalidate — у них та же
   проблема с правками из админки; применить тот же подход.

Критерии приёмки:
- Изменить слово в `pages.body` через админку → в течение ~минуты (после
  обновления страницы) изменение видно на живом сайте без редеплоя.
